## 1. 项目一句话定位

**Prompt OS 是一个面向高频使用 AI 的个人用户的场景驱动 Prompt 工作台，将零散提示词、可复用上下文和分步任务包沉淀为可以查找、调用和分发的个人 AI 工作系统。**

证据：

- 应用界面标语为“场景驱动的提示词工作台”，见 `src/App.tsx`。
- PRD 将产品定义为“个人复利型 AI 提效系统”，强调从收藏转向调用，见 `Prompt_OS_产品需求文档_v0.2.md`。
- 当前主模型已经从早期规划中的 `Workflow` 演进为 `TaskPack` + `TaskStage`，见 `src/types/index.ts` 与 `src/router.tsx`。

## 2. 目标用户与使用场景

### 目标用户

根据 PRD，可确认的目标用户包括：

- 高频使用 AI 的知识工作者。
- 产品经理、创业者、研究者、咨询顾问。
- 内容创作者、自媒体作者、设计师。
- 已经积累大量 Prompt，但缺少整理、检索和复用方式的个人用户。

上述为产品设定的服务对象；是否已有真实用户、用户数量和具体用户反馈均为 **【需要我补充】**。

## 3. 核心问题

### 3.1 AI 经验散落，无法形成资产

用户可能在文档、聊天记录或收藏夹中留下大量提示词，但内容没有统一结构。Prompt OS 将 Prompt、上下文、场景和任务包拆为可管理对象，使内容从文本存档变成可继续整理的资产。

### 3.2 有收藏，但遇到任务时仍不知道调用什么

普通收藏逻辑回答“我保存过什么”；本项目的产品判断是回答“我现在应该从哪里开始”。因此它提供场景入口、关键词搜索和 AI 意图搜索，而不是只做列表收藏。

### 3.3 复杂任务需要步骤，不是单条 Prompt

真实任务往往需要先分析、再生成、再检查。当前代码使用 `TaskPack` 承载解决方案，用 `stages` 与阶段内 `promptIds` 表示推荐顺序，使用户能按步骤执行而不是在大量条目中自行拼接。

### 3.4 重复上下文造成调用成本与输出不稳定

个人背景、项目材料、输出规范等信息需要反复提供。项目将其建模为 `Context`，支持类型、触发策略、作用范围与手动选择，降低重复输入成本，同时保留用户对最终拼接内容的控制。

### 3.5 好的方法论难以完整分发

分享单条文本不足以传递场景、顺序与配套内容。项目提供模板、邀请码、兑换预览和冲突检查链路，使一组结构化内容能够被导入另一个本地工作区。

## 5. 核心功能模块

### 5.1 场景驱动工作台与任务包

**用户价值**

用户从任务场景进入，不必先理解所有 Prompt；进入任务包后可以按照阶段顺序逐项使用内容。

**相关实现**

- 页面与组件：`src/pages/TaskWorkbench.tsx`、`src/components/taskpack/WorkbenchSidebar.tsx`、`src/components/taskpack/TaskPackWorkspace.tsx`、`src/components/taskpack/PromptRunner.tsx`。
- 数据结构：`Scenario`、`TaskPack`、`TaskStage`。
- 数据操作：`src/db/repos/taskPackRepo.ts`、`src/db/repos/scenarioRepo.ts`。
- 关键规则：`TaskPack.sceneCategoryId` 指向一级场景；阶段次序由 `stages` 与 `promptIds` 表达。

**适合写进简历/网站**

> 设计场景驱动的任务包工作台，将多条 Prompt 组织为可排序的阶段流程，使用户按真实任务路径调用 AI 方法资产。**【需确认个人职责】**

### 5.2 Prompt 库、详情与变量运行

**用户价值**

用户可以维护 Prompt 的用途、标签、边界等信息，查看 Markdown 正文；对含插槽的 Prompt 填入变量后复制可用文本，同时仍能复制原始正文。

**相关实现**

- 页面与组件：`src/pages/Prompts.tsx`、`src/components/prompt/PromptEditor.tsx`、`src/components/prompt/PromptDetail.tsx`、`src/components/MarkdownView.tsx`。
- 服务：`src/services/templateVariables.ts`。
- 数据结构：`Prompt` 包含正文、摘要、任务意图、场景、标签、输入要求、输出格式、上下游与使用状态等字段。

**适合写进简历/网站**

> 将 Prompt 从纯文本升级为带场景、用途和调用边界的结构化能力卡片，并提供变量填写与原始 Markdown 复制流程。**【需确认个人职责】**

### 5.3 上下文库与带上下文调用

**用户价值**

用户可以保存角色、项目背景、输出规范等长期片段，调用 Prompt 时选择需要附加的内容并预览拼接结果。

**相关实现**

- 页面与组件：`src/pages/Contexts.tsx`、`src/components/context/ContextEditor.tsx`、`src/components/context/MountedContextChips.tsx`、`src/components/context/SceneDefaultContextPanel.tsx`、`src/components/prompt/UsePromptDialog.tsx`。
- 数据结构：9 种 `ContextType`，三种触发策略 `always/manual/conditional`，三种作用范围 `global/scene_category/sub_scene`。
- 数据迁移：`src/db/migrations/v9.ts` 将旧 Context 类型归一为新模型。

**实现边界**

代码已经实现上下文配置和挂载判断，并在详情中展示可能附加的上下文；“带上下文使用”弹窗最终拼接内容目前主要基于 Prompt 显式绑定、默认项和用户手动选择。简历中应写为“支持规则配置与选择拼接”，不夸写为所有条件均已自动注入最终 Prompt。

**适合写进简历/网站**

> 建立可复用上下文模型和调用预览机制，让用户在重复 AI 任务中复用背景材料，同时保留最终输入控制权。**【需确认个人职责】**

### 5.4 批量导入、AI 分类与待确认

**用户价值**

用户粘贴已有 Prompt 文本后，系统可按空行或 `###` 拆分条目，并调用 AI 补齐分类信息；分类把握不足的条目进入人工确认队列。

**相关实现**

- 页面：`src/pages/Import.tsx`、`src/pages/PendingReview.tsx`。
- 服务：`src/services/parser.ts`、`src/services/classifier.ts`、`src/services/relationBuilder.ts`。
- 规则：分类并发默认为 `3`；结果 `confidence < 0.7` 时设置 `pendingReview`。

**适合写进简历/网站**

> 将历史 Prompt 的拆分、分类和低置信度复核串成导入流程，使 AI 辅助整理具备可检查、可纠错的产品闭环。**【需确认个人职责】**

### 5.5 搜索与 AI 意图推荐

**用户价值**

**相关实现**

**适合写进简历/网站**

> 设计关键词直达与自然语言意图推荐双路径搜索，让用户既能精确查找，也能以任务描述发现可用方法。**【需确认个人职责】**

### 5.6 模板打包、邀请码兑换与创作者后台

**用户价值**

## 6. 产品设计价值

### 信息架构

- 当前用户端主导航收敛为“工作台 / Prompt 库 / 上下文库 / 兑换 / 设置”，将使用入口、资产维护和分发入口分开。
- 核心对象关系为：一级场景承载入口，任务包承载解决方案，阶段承载执行顺序，Prompt 承载可复制的具体能力，上下文承载复用输入。
- 早期 PRD 使用 `Workflow` 概念；当前产品已经用 `TaskPack` 替代其主路径，并将旧路由重定向到工作台。这体现了产品对象可以随实际使用方式迭代。

### 用户流程

1. **直接调用路径**：选择场景 -> 选择任务包 -> 查看阶段 -> 打开 Prompt -> 填变量或带上下文复制。
2. **资产整理路径**：粘贴旧 Prompt -> 拆分 -> AI 分类 -> 低置信度人工确认 -> 可选构建上下游关系。
3. **发现路径**：输入关键词直接筛选，或切换 AI 模式输入意图，获得匹配资产与下一步建议。
4. **分发路径**：后台选内容打包 -> 生成邀请码 -> 用户预览内容 -> 检测冲突 -> 确认兑换到本地。

### 页面结构

- 桌面端以三栏为核心：左侧找入口、中部完成主要工作、右侧查看或运行 Prompt。
- 移动端复用同一套内容组件，通过抽屉和全屏详情适配，不额外复制业务页面。
- Prompt 详情把“正文与复制”置于核心区域，元数据和上下游关系作为辅助信息，符合“最终要使用文本”的任务目标。

### 功能边界与重要取舍

| 产品判断 | 代码/文档依据 | 体现的产品能力 |
|---|---|---|
| 从“保存”转向“调用” | PRD 定位、场景工作台和双搜索入口 | 识别用户真正完成任务的关键时刻 |
| 任务包只挂一级场景，步骤用阶段表达 | `TaskPack.sceneCategoryId` 约束与项目说明 | 避免把目录层级误当作流程，维持信息结构清晰 |
| 原始正文可直接复制 | `PromptDetail`、PRD 验收标准 | 理解下游使用场景，避免展示层污染可执行内容 |
| 低置信度结果进入待确认 | `Import.tsx`、`PendingReview.tsx` | 将 AI 风险转成用户可处理的界面状态 |
| 分发时冲突即拒绝，不覆盖本地内容 | `services/redeem.ts` | 优先保护用户资产与引用一致性 |
| 主资产本地存储，联网能力可选 | Dexie 主数据与 Supabase 分发实现 | 在隐私、可用性与分享能力之间做范围控制 |

### 如何体现 AI 产品经理 / 产品经理能力

- 能从“我有很多 Prompt”抽象出调用、编排、上下文和分发四类产品问题。
- 能将 AI 的不确定性设计成可见的复核流程，而非只展示“智能”卖点。
- 能在概念演进中用 `TaskPack` 替代早期 `Workflow`，说明以用户使用路径校正产品模型的能力。
- 能把产品判断落实为页面、字段、状态和约束，而不仅是 PRD 描述。上述归属均为 **【需确认】**。

- `pages/` 负责页面级流程，`components/` 按 prompt、taskpack、context、layout 等业务区域拆分。
- `db/` 提供 Dexie 表、迁移、seed、快照与 repo 函数；主要业务对象由 IndexedDB 持久化。
- `services/` 封装 AI、搜索、模板、兑换、备份等跨页面能力。
- `store/` 中设置持久化；临时选择、搜索、弹窗与反馈状态留在内存。
- 常规资产读写流为：页面/组件 -> repo 或查询 hook -> Dexie -> IndexedDB。
- AI 流为：页面/组件 -> service -> `callJSON()` -> Anthropic API -> 结构化结果 -> 本地表或界面状态。

### 数据模型与演进

- 本地数据库定义 `prompts`、`contexts`、`scenarios`、`taskPacks`、`sharedTemplates`、`workflows` 六张表，其中 `workflows` 为兼容保留对象。
- 当前 schema 版本为 v9：v6 引入 `TaskPack`，v7 切换 v3.1 出厂内容，v8 为任务包补排序逻辑，v9 重构上下文类型与触发字段。
- v7 与 v9 在迁移前保存归档；项目说明明确记录过早期破坏性变更造成的数据丢失教训。这是一项可讲述的工程改进过程，而非“从未出错”的结果。
- 启动时采用幂等 seed：v3.1 内容和“AI 编程·从 0 到 1”补充内容各自通过固定 probe ID 避免重复导入。

### 状态、接口与权限逻辑

- 关键词搜索逻辑被集中到 `src/services/keywordMatch.ts`，防止各页面各写一套匹配规则。
- 全局提示和确认弹窗通过 Zustand store + 根组件呈现，避免各处直接调用浏览器原生弹窗。
- AI Key 与模型配置由用户在设置页输入并保存在本机 localStorage；这适合个人本地工具，但不等于生产 SaaS 的密钥托管方案。
- 后台采用 Supabase Auth 登录并通过 `admins` 表校验权限；模板、邀请码与兑换记录受 RLS 管理。
- 匿名用户不能直接读写业务表，只通过 `preview_invite_code` 与 `redeem_invite_code` RPC 获取被允许的兑换能力。

## 12. 面试讲述版本

### 30 秒版本

> 我做了一个叫 Prompt OS 的项目 **【需确认角色范围】**。它解决的是高频使用 AI 的人积累了很多 Prompt，却在真正做任务时仍然找不到、不会组合、还要反复补背景信息的问题。产品把内容组织成场景、任务包、Prompt 和上下文：用户从任务场景进入，按阶段调用 Prompt，也可以用关键词或 AI 意图搜索寻找资产。当前代码内置 18 个场景、171 条 Prompt 和 102 个任务包；AI 还用于导入分类、关系构建和标题建议。主数据存在本地，模板分发通过邀请码与 Supabase 完成。

### 2 分钟版本

> Prompt OS 的起点不是“做一个 Prompt 收藏夹”，而是一个更具体的问题：当一个人积累了很多 AI 使用经验后，如何在下一次真实任务发生时快速调用它们。PRD 把痛点归纳为内容散落、难以查找、难以组合、重复输入上下文，以及方法论难分享。
>
> 当前产品的核心模型是场景、任务包、Prompt 和上下文。场景是入口；任务包是一个可执行的解决方案，里面有按顺序排列的阶段；Prompt 是可以填写变量、拼接上下文并复制给 AI 的最小能力单元。这里有一个重要演进：早期规划使用 Workflow，当前实现已经改成 TaskPack，旧路由也重定向回工作台，因为用户更容易理解“我进入一个任务包跟着做”，而不是先搭一个抽象流程。
>
> AI 并不是独立的聊天功能，而是进入了整理和调用流程：导入页会拆分文本并调用 AI 分类，置信度低于 0.7 的结果进入待确认；搜索框既能做本地关键词检索，也能按自然语言意图推荐 Prompt、任务包和上下文；设置页可以让 AI 重建 Prompt 的上下游关系。这个设计的重点是让 AI 加速整理，同时保留人工复核。
>
> 工程上，主数据存在 Dexie/IndexedDB 中，当前 schema 到 v9；v7 和 v9 已加入迁移归档和快照恢复，用来降低本地升级风险。模板分发走 Supabase：后台管理模板和邀请码，用户先预览内容并检查本地冲突，再通过 RPC 兑换，后端用权限规则和行锁限制越权及重复兑换。可证明的交付产物包括应用代码、PRD、出厂内容和测试；真实用户效果、我的具体职责和发布状态需要用实际经历补充。

### 面试官可能追问的 8 个问题

1. **为什么这不是一个普通的 Prompt 收藏夹？**  
   回答锚点：工作台以场景和任务包驱动调用；Prompt 详情支持变量与上下文；搜索提供意图推荐。证据见 `TaskWorkbench.tsx`、`TaskPackWorkspace.tsx`、`searcher.ts`。

2. **为什么从 `Workflow` 改为 `TaskPack`？**  
   回答锚点：现行数据模型和路由已经将任务包作为用户可进入的解决方案，旧 Workflow 仅兼容保留；强调用更贴近用户任务语言的对象替代早期抽象。

3. **AI 分类出错怎么办？阈值为什么是 0.7？**  
   回答锚点：当前实现把低置信度结果放进待确认页；`0.7` 是代码中的产品规则，阈值效果是否经过用户验证为 **【需要我补充】**。

4. **为何选本地存储，而不是从第一天做云同步？**  
   回答锚点：核心个人资产在 IndexedDB，可在不依赖账号体系的情况下使用；联网只承担 AI 和分发。多设备同步是否规划为 **【需要我补充】**。

5. **本地数据升级如何避免用户资产受损？**  
   回答锚点：如实讲历史教训，以及 v7/v9 归档与快照恢复实现；不要声称所有历史升级都没有风险。证据见 `src/db/index.ts`、`src/db/snapshot.ts`。

6. **上下文规则是否已经完全自动生效？**  
   回答锚点：类型、触发策略、范围和挂载计算已实现；最终拼接预览仍以显式绑定、默认及手选为核心，需要按当前实现描述，不扩大范围。

7. **邀请码分发如何保护权限和本地内容？**  
   回答锚点：用户端先 `preview` 并做本地 ID 冲突检查，确认后再 `redeem`；Supabase RLS 与 RPC 控制访问，兑换 RPC 用行锁。证据见 `redeem.ts` 与 `schema.sql`。

8. **项目能否作为 AI 教育或课程设计案例？**  
   回答锚点：“AI 编程·从 0 到 1”已经形成分阶段任务包，具备学习路径形式；但是否真正用于教学、受众和结果必须以真实材料补充，不能仅凭代码断言。
