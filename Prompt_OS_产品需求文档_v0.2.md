# Prompt OS 产品需求文档

> 版本：v0.2  
> 文档类型：产品需求文档（PRD）  
> 产品定位：个人复利型 AI 提效系统  
> 核心目标：帮助用户将提示词、上下文、场景和工作流沉淀为可检索、可复用、可组合、可分享的个人资产库。

---

## 1. 产品概述

### 1.1 产品名称

暂定名称：**Prompt OS**

### 1.2 产品一句话描述

Prompt OS 是一个帮助用户把提示词、上下文和工作流沉淀为个人复利资产的 AI 提效系统。

### 1.3 产品定位

Prompt OS 面向高频使用 AI 的个人创作者、知识工作者、创业者、产品经理、研究者、内容创作者和效率工具爱好者。

它不是一个普通的 Prompt 收藏夹，而是一个用于管理、调用和组合 AI 能力的个人工作系统。

用户可以将分散在文档、聊天记录、收藏夹中的提示词统一整理，并按照真实场景、任务类型、上下游关系和工作流进行结构化管理。最终，用户可以通过输入需求、任务、关键词或意图，快速找到合适的 Prompt、上下文和工作流组合。

---

## 2. 产品背景

随着 AI 工具的普及，越来越多用户积累了大量提示词。但这些提示词通常存在以下问题：

1. 分散在不同文档、聊天记录、收藏夹或笔记工具中。
2. 想用时找不到。
3. 不知道哪条 Prompt 适合当前任务。
4. 多条 Prompt 之间缺乏协作关系。
5. 每次使用 AI 都需要重复输入个人背景、任务信息和输出偏好。
6. 很难将自己的方法论整理成可分享、可复刻的系统。

普通用户每次都从零开始使用 AI，而拥有 Prompt 库、上下文库和工作流系统的用户，能够让每次使用都成为下一次的基础。

本产品要解决的核心问题是：

**如何让用户把一次性的 AI 使用经验，沉淀为可复用、可组合、可分享的长期资产。**

---

## 3. 产品目标

### 3.1 核心目标

建立一套可复利增长的个人 AI 提效框架。

具体包括：

1. 建立个人 Prompt 库。
2. 建立场景化任务库。
3. 建立可复用上下文库。
4. 建立 Prompt 之间的协作工作流。
5. 支持 AI 根据用户意图推荐合适提示词。
6. 支持将整理好的框架模板分享给他人复刻使用。
7. 支持批量整理用户已有的 500+ 条提示词，并完成分类、标注和结构化管理。

### 3.2 产品价值

用户面对一个任务时，不需要重新思考“我该怎么问 AI”，而是可以快速调用已有系统。

产品的最终价值不是“保存 Prompt”，而是帮助用户建立一套可持续增强的个人 AI 工作系统。

### 4.2 用户特征

核心用户通常具备以下特征：

3. 希望将 AI 使用经验沉淀成长期资产。
4. 需要在不同场景中快速调用合适 Prompt。

## 5. 核心产品理念

### 5.1 从收藏到调用

普通 Prompt 工具解决的是“我存过什么”。  
Prompt OS 要解决的是“我现在遇到这个任务，应该调用什么”。

### 5.2 从单条 Prompt 到工作流

复杂任务通常不能靠单条 Prompt 完成。  
产品需要支持多条 Prompt 按照上游、下游、执行顺序组合成完整任务流程。

### 5.3 从文本到资产

每条 Prompt 不只是文本，而是一个可复用能力单元。  
它应该拥有标题、场景、任务类型、输入要求、输出结果、上下游关系、适用边界和关键词标签。

### 5.4 从工具到系统

工具是别人的，系统是自己的。  
本产品的目标是帮助用户建立自己的 AI 工作系统，而不是依赖一次性的 AI 对话。

## 6. 核心对象模型

产品中包含四类核心对象：

1. Prompt
2. Context
3. Scenario
4. Workflow

这四类对象共同构成用户的个人 AI 工作系统。

---

### 6.1 Prompt：最小能力单元

Prompt 是系统中的最小能力单元。

每条 Prompt 应该包含：

| 字段 | 说明 |
|---|---|
| Prompt 标题 | 用一句话说明这条 Prompt 的用途 |
| Prompt 正文 | 保留原始提示词内容，支持一键复制 |
| 所属场景 | 该 Prompt 适合在哪些场景下使用 |
| 任务类型 | 分析、生成、改写、总结、规划、决策、复盘等 |
| 输入要求 | 使用前需要用户提供哪些信息 |
| 输出结果 | 该 Prompt 通常会生成什么内容 |
| 上游 Prompt | 使用它之前推荐先调用的 Prompt |
| 下游 Prompt | 使用它之后推荐继续调用的 Prompt |
| 推荐组合 | 适合一起使用的其他 Prompt |
| 适用边界 | 哪些情况下不适合使用 |
| 关键词标签 | 用于搜索和推荐 |
| 难度等级 | 初级、中级、高级 |
| 价值等级 | 高频、高价值、战略型、辅助型等 |
| 是否可分享 | 是否允许被打包进模板分享 |

关键要求：

**Prompt 正文必须独立存储。复制时只复制正文，不复制标签、说明或备注。**

---

### 6.2 Context：上下文信息

Context 用于保存用户长期复用的信息。

上下文类型包括：

1. 个人背景。
2. 职业身份。
3. 长期目标。
4. 当前项目。
5. 表达风格。
6. 输出偏好。
7. 常用限制条件。
8. 目标用户画像。
9. 产品或业务介绍。
10. 知识背景。
11. 常用素材，如简历、个人介绍、品牌介绍、项目说明等。

使用方式：

用户调用 Prompt 时，可以选择一个或多个上下文自动拼接。

示例：

用户选择：
- 个人定位
- 当前项目背景
- 写作风格

然后调用：
- 文章生成 Prompt

系统自动将这些上下文组合进任务中，减少重复输入。

---

### 6.3 Scenario：场景库

Scenario 是用户进入系统的主要入口。

用户通常不是按照 Prompt 名称找内容，而是按照真实任务场景寻找解决方案。场景库应采用树状结构组织，避免一级目录下堆积大量 Prompt。

推荐结构：

```text
一级场景 > 二级场景 > 三级任务 > Prompt 名称
```

默认一级场景建议控制在 8 到 12 个，初始可包含：

1. 工作场景
2. 自媒体场景
3. 学习场景
4. 读书场景
5. 写作场景
6. 商业场景
7. 个人成长场景
8. 生活管理场景
9. AI 工具场景
10. 知识管理场景

每个场景下可以包含：

1. 推荐 Prompt。
2. 推荐工作流。
3. 常用上下文。
4. 典型任务。
5. 使用案例。
6. 适用人群。
7. 推荐使用路径。

---

### 6.4 Workflow：Prompt 协作流程

Workflow 用于将多条 Prompt 编排成完整任务流程。

每个工作流包含：

| 字段 | 说明 |
|---|---|
| 工作流名称 | 工作流的名称 |
| 适用目标 | 该工作流解决什么问题 |
| 包含 Prompt | 该流程中使用的所有 Prompt |
| 执行顺序 | Prompt 的使用顺序 |
| 每一步输入 | 每一步需要用户提供的信息 |
| 每一步输出 | 每一步会产出的内容 |
| 下一步动作 | 每一步完成后建议继续做什么 |
| 最终交付物 | 整个工作流最终生成什么 |
| 适用边界 | 哪些情况下不适合使用该工作流 |

示例工作流：深入学习一个新知识

1. 学习目标澄清 Prompt
2. 知识地图生成 Prompt
3. 核心概念拆解 Prompt
4. 案例和类比生成 Prompt
5. 误区与反共识分析 Prompt
6. 行动计划生成 Prompt

## 7. 核心使用场景

### 7.1 根据任务快速找到 Prompt

用户输入：

> 我想系统学习一个新领域。

系统返回：

1. 推荐 Prompt：知识地图生成 Prompt。
2. 推荐工作流：深度学习五步工作流。
3. 推荐上下文：个人学习偏好、已有知识水平。
4. 输出建议：先生成框架，再逐层深入。

目标：

让用户不需要知道 Prompt 名称，也能通过自然语言需求找到合适工具。

---

### 7.2 整理已有 Prompt 文档

用户上传或粘贴 500+ 条提示词。

系统自动完成：

1. 场景分类。
2. 任务类型标注。
3. 上游 Prompt 标注。
4. 下游 Prompt 标注。
5. 推荐组合关系。
6. 适用边界说明。
7. 关键词标签生成。
8. 使用说明生成。

关键要求：

**所有标注信息必须和 Prompt 正文分离，不能影响用户复制原始 Prompt。**

---

### 7.3 组合多个 Prompt 完成复杂任务

用户选择一个目标：

> 深入研究一个商业机会。

系统推荐工作流：

1. 行业背景分析 Prompt。
2. 用户痛点挖掘 Prompt。
3. 竞品分析 Prompt。
4. 商业模式推演 Prompt。
5. 风险与反共识分析 Prompt。
6. 行动计划生成 Prompt。

每一步都应包含：

1. 当前步骤目标。
2. 所需输入。
3. 推荐 Prompt。
4. 预期输出。
5. 下一步建议。

---

### 7.4 复用个人上下文

用户提前保存个人背景：

1. 我是谁。
2. 我现在的目标。
3. 我的项目。
4. 我的表达风格。
5. 我的长期任务。
6. 我的常用限制条件。

之后调用 Prompt 时，系统可以自动关联这些上下文。

示例：

用户要写文章，系统自动带入：

> 用户偏好结构清晰、结论前置、少废话、有行动建议的表达风格。

目标：

减少重复输入，让用户的个性化背景成为长期可复用资产。

---

### 7.5 分享人生框架模板

用户可以将自己的场景库、Prompt 库、上下文模板和工作流打包成一个模板。

其他用户导入后，可以直接获得一整套可使用的 AI 提效系统。

模板内容包括：

1. 场景分类。
2. Prompt 列表。
3. 工作流组合。
4. 上下文模板。
5. 使用说明。
6. 推荐使用路径。
7. 适用人群。
8. 适用边界。

## 11. Prompt 整理标准

用户后续会提供 500+ 条提示词。系统需要对这些 Prompt 进行结构化整理。

### 11.1 每条 Prompt 的整理格式

```text
【标题】
一句话概括这条 Prompt 的用途。

【Prompt 正文】
这里保留原始提示词，保证可直接复制使用。

【所属场景】
例如：学习成长 / 写作表达 / 商业分析 / 个人复盘。

【任务类型】
例如：分析、生成、改写、总结、决策、规划、复盘。

【适合什么时候用】
说明它在什么情况下最有价值。

【输入要求】
用户使用前需要提供哪些信息。

【输出结果】
这条 Prompt 通常会生成什么。

【上游 Prompt】
使用它之前推荐先用哪些 Prompt。

【下游 Prompt】
使用它之后推荐接哪些 Prompt。

【推荐组合】
适合和哪些 Prompt 组成工作流。

【适用边界】
哪些情况不适合使用，或者容易失效。

【关键词标签】
方便检索的标签。
```

### 11.2 重要原则

1. 不修改原始 Prompt 的核心含义。
2. Prompt 正文必须保持可复制。
3. 标注信息不得混入 Prompt 正文。
4. 分类要服务于检索和调用，不是为了形式整齐。
5. 上游 / 下游关系要基于真实任务流程，而不是机械排序。
6. 适用边界必须明确，避免用户误用。
7. 同一条 Prompt 可以属于多个场景。
8. 同一条 Prompt 可以被多个工作流复用。

## 12. 初始导入后的分类与导航体系

### 12.1 设计目标

用户导入大量 Prompt 后，系统需要先完成初始化分类，让 Prompt 从“零散文本”变成“可浏览、可检索、可推荐、可组合”的结构化资产。

分类的首要目标不是把目录做得整齐，而是让用户在真实任务发生时，能快速定位可用 Prompt。

核心原则：

**左侧目录负责浏览，搜索框负责直达，AI 推荐负责决策。**

---

### 12.2 左侧边栏结构

左侧边栏建议采用三段式结构：

```text
快捷入口
  最近使用
  我的收藏
  高频工作流
  待分类 Prompt

场景库
  工作场景
  自媒体场景
  学习场景
  读书场景
  写作场景
  商业场景
  个人成长场景
  生活管理场景
  AI 工具场景
  知识管理场景

智能集合
  适合写作的 Prompt
  适合分析的 Prompt
  适合决策的 Prompt
  适合复盘的 Prompt
  适合深度研究的 Prompt
```

场景库采用四层结构：

```text
一级场景：用户所在领域
二级场景：具体使用场景
三级任务：当前要完成的动作
四级内容：Prompt 名称
```

示例：

```text
工作场景
  汇报表达
    周报月报
      - 生成周报
      - 优化周报表达
      - 提炼工作亮点
      - 生成下周计划
```

---

### 12.3 分类颗粒度规则

为避免用户再次陷入“大海捞针”，分类需要遵守以下规则：

1. 一级场景建议控制在 8 到 12 个。
2. 最末级目录下建议不超过 8 条 Prompt，最多不超过 12 条。
3. 如果某个末级目录超过 12 条，应继续拆分三级任务。
4. 目录名称应使用用户真实任务语言，避免过度抽象。
5. Prompt 可以属于多个场景，但数据库中只保留一条 Prompt 记录，通过引用关系挂载到多个目录。

推荐命名方式：

```text
生成会议纪要
优化小红书标题
拆解复杂概念
分析用户痛点
制定学习计划
```

不推荐命名方式：

```text
万能提示词
高级分析助手
效率神器
内容优化工具
```

---

### 12.4 Prompt 的主场景与副场景

每条 Prompt 导入后，必须被分配一个主场景、一个主任务，同时允许拥有多个副场景和多个标签。

```text
主场景：它最常被使用的地方
副场景：它也可能有用的地方
标签：它可以被横向检索到的关键词
```

示例：

```text
Prompt：反共识分析 Prompt

主场景：
商业场景 > 决策分析 > 反共识判断

副场景：
学习场景 > 深度研究
工作场景 > 战略分析
个人成长场景 > 重大选择

标签：
反共识、决策、风险、战略、深度分析、多视角
```

这样既能保持目录秩序，又能避免 Prompt 被单一文件夹限制。

---

### 12.5 初始化分类流程

用户批量导入 Prompt 合集后，系统按以下流程处理：

#### 第一步：清洗与切分

识别每条 Prompt 的边界，去除重复空行、混乱编号和无效标题，将整份文档拆分为独立 Prompt 条目。

#### 第二步：生成标准标题

为缺少标题或标题含糊的 Prompt 生成短标题。

标题格式建议：

```text
动词 + 任务对象
```

例如：

```text
生成会议纪要
提炼文章观点
制定学习计划
分析竞品策略
```

#### 第三步：自动场景分类

系统为每条 Prompt 分配：

1. 一级场景。
2. 二级场景。
3. 三级任务。
4. 主场景。
5. 副场景。
6. 分类置信度。

分类置信度低于阈值的 Prompt 进入“待分类 Prompt”或“待确认 Prompt”。

#### 第四步：生成标签

标签用于横向检索，不替代目录。

例如小红书改写类 Prompt 可以拥有：

```text
小红书、标题、正文、改写、情绪价值、内容生产、平台适配
```

#### 第五步：识别上下游关系

系统判断该 Prompt 在任务链条中的位置。

示例：

```text
上游 Prompt：明确目标用户、分析内容选题、拆解用户痛点
当前 Prompt：生成小红书正文
下游 Prompt：优化标题、生成封面文案、检查平台风格、生成发布复盘
```

#### 第六步：生成工作流推荐

系统根据 Prompt 之间的协作关系，自动推荐可复用工作流。

示例：

```text
工作流：小红书笔记生产流程

1. 明确目标用户
2. 生成选题方向
3. 判断选题传播潜力
4. 生成标题
5. 生成正文
6. 生成封面文案
7. 检查平台风格
8. 生成发布复盘维度
```

---

### 12.6 初始化分类字段

每条 Prompt 初始化后，至少需要生成以下字段：

| 字段 | 说明 |
|---|---|
| Prompt ID | 唯一标识 |
| Prompt 标题 | 便于浏览和检索的短标题 |
| Prompt 正文 | 原始提示词，保持可直接复制 |
| 一级场景 | 所属领域，如工作场景、自媒体场景 |
| 二级场景 | 具体使用场景，如会议协作、内容生产 |
| 三级任务 | 当前动作，如生成会议纪要、优化标题 |
| 主场景路径 | 该 Prompt 最主要的使用路径 |
| 副场景路径 | 该 Prompt 可复用的其他场景 |
| 任务意图 | 该 Prompt 解决的真实问题 |
| 输入要求 | 使用前需要提供的信息 |
| 输出结果 | 通常产出的内容形态 |
| 关键词标签 | 用于横向搜索和智能推荐 |
| 上游 Prompt | 使用前建议先调用的 Prompt |
| 下游 Prompt | 使用后建议继续调用的 Prompt |
| 推荐工作流 | 可加入的工作流 |
| 适用边界 | 不适合使用的情况 |
| 分类置信度 | AI 自动分类的可信程度 |
| 是否需要人工确认 | 低置信度或多义 Prompt 需要确认 |

---

### 12.7 推荐页面布局

Prompt 库核心页面建议采用三栏布局：

```text
左侧：场景树目录
中间：Prompt 卡片列表
右侧：Prompt 详情、上下游推荐、一键复制
顶部：AI 检索框
```

用户可以通过三种方式找到 Prompt：

1. 浏览左侧场景树。
2. 通过关键词或标签搜索。
3. 输入自然语言任务，让 AI 推荐 Prompt 和工作流。

示例：

用户输入：

```text
我要准备一次重要汇报
```

系统应返回：

```text
推荐场景：
工作场景 > 汇报表达 > 重要汇报

推荐 Prompt：
- 梳理汇报目标
- 提炼核心结论
- 生成汇报大纲
- 优化老板视角表达
- 预测可能被问到的问题

推荐工作流：
重要汇报准备流程
```

### 12.8 验收标准

初始化分类功能的验收标准：

1. 导入后，每条 Prompt 都能被拆分为独立条目。
2. 每条 Prompt 都有主场景、三级目录路径和关键词标签。
3. 最末级目录下 Prompt 数量原则上不超过 12 条。
4. Prompt 正文与分类标注完全分离。
5. 同一条 Prompt 可以被多个场景引用，但不重复存储。
6. 低置信度分类会进入待确认列表。
7. 用户输入自然语言任务后，系统能推荐对应 Prompt、场景和工作流。

## 14. MVP 版本

第一版不应追求功能复杂，而应验证核心价值：

**用户能不能比以前更快找到并使用合适的 Prompt。**

### 13.1 MVP 功能范围

1. Prompt 导入  
   支持批量导入已有提示词。

2. 自动分类  
   AI 自动识别每条 Prompt 的场景、任务和标签。

3. Prompt 卡片  
   每条 Prompt 以卡片展示，支持正文复制。

4. 搜索与推荐  
   用户输入任务，系统推荐相关 Prompt。

5. 工作流组合  
   支持把多条 Prompt 手动或半自动组合成任务流程。

6. 上下文模板  
   支持保存常用个人背景和任务信息。

7. 模板导出 / 分享  
   支持将某个场景或工作流打包分享。

## 15. MVP 验收标准

### 14.1 Prompt 导入验收

用户可以一次性导入多条 Prompt，系统能够将其拆分成独立条目。

验收标准：

1. 支持批量导入。
2. 能识别每条 Prompt 的边界。
3. 导入后每条 Prompt 都可以单独编辑和复制。

---

### 14.2 自动分类验收

系统可以为每条 Prompt 自动生成基础元数据。

验收标准：

1. 自动识别所属场景。
2. 自动识别任务类型。
3. 自动生成关键词标签。
4. 自动生成适用说明。
5. 自动建议上游 / 下游 Prompt。

---

### 14.3 搜索推荐验收

用户输入自然语言任务后，系统可以推荐合适 Prompt。

验收标准：

1. 支持关键词搜索。
2. 支持意图搜索。
3. 推荐结果包含 Prompt、场景和工作流。
4. 推荐结果解释为什么适合当前任务。

---

### 14.4 Prompt 复制验收

用户可以一键复制 Prompt 正文。

验收标准：

1. 复制内容只包含 Prompt 正文。
2. 不包含标题、标签、备注、边界说明等元数据。
3. 复制操作足够明显且低干扰。

---

### 14.5 工作流验收

用户可以创建并使用由多条 Prompt 组成的工作流。

验收标准：

1. 可以添加多个 Prompt 步骤。
2. 可以调整步骤顺序。
3. 每一步有输入说明和输出说明。
4. 可以查看完整流程。
5. 可以复制单步 Prompt 或完整工作流。

## 17. 风险与边界

### 16.1 风险：Prompt 变成新的信息垃圾

如果只导入、不分类、不调用，Prompt 库会变成新的收藏夹。

解决方式：

1. 强制结构化元数据。
2. 通过场景和任务驱动检索。
3. 鼓励将高价值 Prompt 组合成工作流。

---

### 16.2 风险：分类过细导致使用成本升高

分类太复杂会让用户不知道放在哪里。

解决方式：

1. 一级分类保持克制。
2. 支持一条 Prompt 属于多个场景。
3. 用标签补充细分，不用层级无限嵌套。

---

### 16.3 风险：工作流过重

如果每次使用都需要配置复杂流程，用户会放弃。

解决方式：

1. 支持单条 Prompt 快速调用。
2. 支持轻量工作流。
3. 工作流可以分步运行，而不是强制一次性执行。

---

### 16.4 风险：上下文污染输出

上下文过多可能导致 AI 输出偏离任务。

解决方式：

1. 用户手动选择上下文。
2. 系统推荐上下文，但不强制绑定。
3. 每个上下文有明确类型和适用场景。
4. 支持临时关闭默认上下文。

---

## 18. 成功指标

### 17.1 用户效率指标

1. 用户找到合适 Prompt 的时间缩短。
2. 用户重复输入背景信息的次数减少。
3. 用户使用工作流完成复杂任务的比例提高。

### 17.2 内容资产指标

1. Prompt 总数。
2. 已分类 Prompt 占比。
3. 已绑定场景 Prompt 占比。
4. 已加入工作流 Prompt 占比。
5. 上下游关系完整度。
6. 模板导出数量。

### 17.3 使用行为指标

1. 搜索次数。
2. Prompt 复制次数。
3. 工作流运行次数。
4. 上下文调用次数。
5. 模板分享次数。
6. 模板导入次数。

---

## 19. 总结

Prompt OS 的核心不是保存提示词，而是帮助用户建立自己的 AI 工作系统。

它解决的不是“我有多少 Prompt”，而是：

1. 当前任务应该用哪条 Prompt？
2. 这条 Prompt 前后应该连接什么？
3. 应该带入哪些上下文？
4. 有没有一套完整工作流可以直接使用？
5. 这套方法能不能沉淀下来，下次复用，甚至分享给别人？

最终目标：

**让用户从一次性使用 AI，升级为持续复利地使用 AI。**
